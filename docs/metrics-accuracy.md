# Точность показателей выгрузки (совпадение с Grafana)

## Реализованные настройки (application.properties)

| Параметр | По умолчанию | Описание |
|----------|--------------|----------|
| `victoriametrics.subquery-step` | `1m` | Шаг сэмплирования в подзапросах `avg_over_time` / `max_over_time`. **Сделайте равным Min step в панели Grafana** (например 1m, 15s), чтобы интервал и разрешение совпадали. |
| `victoriametrics.cpu-rate-window` | `5m` | Окно для `rate(container_cpu_usage_seconds_total[X])`. В Grafana обычно 5m или 2m — выставьте такое же. |
| `victoriametrics.aggregation-method` | `sum_then_percent` | Как считаем % по нескольким подам одного контейнера: **sum_then_percent** = sum(usage)/sum(limit), как в панелях Grafana с группировкой по deployment/container; **average_per_pod** = среднее от процентов по каждому поду. |

## Дополнительные варианты (если расхождение остаётся)

1. **Точка оценки времени**  
   Сейчас все instant-запросы выполняются с `time = конец интервала` (как `${__to}` в Grafana). Если в дашборде используется середина интервала или `now`, можно добавить опцию `victoriametrics.evaluation-offset` (например `0.5` = середина) и вычислять `time = from + (to - from) * offset`.

2. **Окно rate для коротких интервалов**  
   При интервале выгрузки меньше 5 минут окно `rate(...[5m])` может обрезаться. Имеет смысл ограничивать окно сверху длиной интервала (уже можно добиться, задав в Grafana тот же интервал и Min step).

3. **Память: другой источник**  
   Сейчас используется `container_memory_working_set_bytes`. Если в Grafana используется `container_memory_rss` или `container_memory_usage_bytes`, нужно добавить настройку выбора метрики памяти и использовать её в запросах.

4. **Отображение дробных %**  
   В таблице проценты округляются до целых. Для уменьшения визуального расхождения с Grafana можно выводить один знак после запятой (потребуется изменить DTO/HTML).

5. **Поды на границе интервала**  
   Количество подов и привязка pod→owner берутся в момент `to`; использование — за весь интервал. Если под перезапускался, возможна небольшая несогласованность. Обычно это даёт малый вклад в расхождение.

## Рекомендация

Сначала подгоните под свой дашборд Grafana:

- **Min step** панели → `victoriametrics.subquery-step`
- Окно **rate** в запросе Grafana → `victoriametrics.cpu-rate-window`
- Оставьте `victoriametrics.aggregation-method=sum_then_percent`, если в Grafana используется агрегация по deployment/container (sum, затем процент от лимита).

После этого расхождение обычно становится минимальным.
