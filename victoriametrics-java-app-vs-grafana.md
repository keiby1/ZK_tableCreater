# Почему результат из приложения (Java + REST API) отличается от Grafana и UI VictoriaMetrics

Кратко — что чаще всего ломает совпадение между приложением и Grafana/UI при «одном и том же» запросе.

---

## 1. Разный тип запроса: instant vs range

- **`/api/v1/query`** (instant) — одна точка во времени. Один момент `time`, одна пара `[timestamp, value]` в ответе.
- **`/api/v1/query_range`** — серия точек за период (`start`, `end`, `step`).

Если в приложении вызывается **instant**, а в Grafana на графике — **range** (или наоборот), вы сравниваете «одно мгновение» и «усреднение по интервалу» — числа будут сильно отличаться.

Для «среднего за интервал» обычно нужен либо:
- **range-запрос** и усреднение по полученным точкам в коде, либо
- **instant-запрос** с выражением, которое уже считает среднее по времени, например:
  `avg_over_time(rate(container_cpu_usage_seconds_total{...}[5m])[30m:])` в момент `time`.

Проверьте: тем же URL и телом запроса, что в приложении, дергаете **тот же endpoint** (query vs query_range) и с теми же `time` / `start`/`end`/`step`.

---

## 2. Как берётся одно число из ответа

Ответ Prometheus/VictoriaMetrics:

- **Instant:** `data.result[].value` — массив из двух элементов `[timestamp, "value"]`.
- **Range:** `data.result[].values` — массив пар `[timestamp, "value"]`.

Типичные ошибки в приложении:

- Берётся **первая** точка вместо **средней** по всем точкам за интервал.
- Берётся **timestamp** вместо **value** (перепутаны элементы пары).
- Есть несколько рядов (`result` из нескольких элементов — например, по подам), а берётся только первый или один, тогда как в Grafana на графике — сумма/среднее по всем рядам.
- Не приводится строка к числу или берётся не тот тип (например, целое вместо float).

Итог: даже при правильном запросе число «в приложении» будет другим, если вы по-другому агрегируете точки или ряды (первая точка vs среднее, один pod vs все).

---

## 3. Время оценки (time window)

- В **Grafana** время задаётся диапазоном панели (и опцией «Resolution» / шаг).
- В **приложении** вы задаёте время параметрами запроса: для instant — `time`, для range — `start`, `end`, `step`.

Если приложение, например, всегда шлёт `time=now` (момент запроса), а в Grafana смотрите «последние 5 минут» с другим концом периода — данные за разные окна, значения не совпадут. То же при расхождении часов (UTC vs локальный сервер Grafana).

Проверьте: в логах/дебаге приложения зафиксировать реальные `time` или `start`/`end`, которые уходят в VM, и в Grafana/UI выставить тот же диапазон и шаг.

---

## 4. Заголовки и параметры «интервала»

Если «интервал передаётся в хедерах» — важно, что именно понимает под этим VictoriaMetrics. Обычно интервал задаётся не кастомным заголовком, а:

- для **query**: параметр `time` (момент);
- для **query_range**: параметры `start`, `end`, `step`.

Если вы, например, в заголовке передаёте «длину окна» (например 5m), а сам запрос идёт без `start`/`end`/`step` или с другими значениями, бэкенд может игнорировать заголовок и использовать свои дефолты. Тогда приложение и Grafana запрашивают разные временные окна — отсюда расхождение.

Имеет смысл в коде логировать итоговые query params и body, которые реально уходят в VM, и сравнить с тем, что видно в Network вкладке в Grafana при том же запросе.

---

## 5. Краткий чеклист

| Что проверить | Действие |
|---------------|----------|
| Один и тот же endpoint | В приложении вызывается тот же URL (query или query_range), что и в Grafana/UI. |
| Одинаковое время | Те же `time` (instant) или `start`/`end`/`step` (range), что на графике. |
| Один и тот же PromQL | Текст запроса без отличий (пробелы, экранирование). |
| Парсинг ответа | Берётся именно `value` (строка числа), не timestamp; при нескольких рядах — решено, как агрегировать (сумма/среднее по рядам). |
| Агрегация по времени | Для range — если нужна «одна цифра за интервал», в коде считается среднее (или последнее) по `result[].values`, а не первая точка. |
| «Интервал» в запросе | Интервал задаётся параметрами API (`start`/`end`/`step` или `time`), а не только заголовком; заголовок не переопределяет окно по-своему. |

Чаще всего в такой ситуации «приложение сильно отличается, Grafana и UI близки» виноваты: **другой тип запроса (instant vs range)**, **другое окно времени** или **неправильное извлечение/агрегация значения из ответа** (одна точка вместо среднего, один series вместо всех). Имеет смысл вывести из приложения фактический URL, метод, query params и один полный JSON-ответ и сравнить с запросом из Grafana по тому же запросу.
